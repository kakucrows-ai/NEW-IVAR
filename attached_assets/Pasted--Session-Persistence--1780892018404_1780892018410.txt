تصرف كمهندس بنية تحتية رئيسي ومتخصص في استمرارية الجلسات (Session Persistence) وأنظمة إعادة التشغيل عالية الاعتمادية لبوتات Messenger.

المشكلة الحالية:
عند إعادة تشغيل البوت (Restart)، يتم فقدان الجلسة أو تلفها أو يتم تسجيل الخروج، مما يؤدي إلى الحاجة لإعادة تسجيل الدخول أو إعادة إنشاء ملفات الكوكيز والجلسات.

مهمتك هي إعادة تصميم نظام إعادة التشغيل بالكامل بحيث يتمكن البوت من إعادة التشغيل بأمان دون فقدان:
- الكوكيز
- ملفات appstate
- الجلسات
- رموز المصادقة
- حالة الاتصال

يجب أن تكون عملية إعادة التشغيل آمنة وقادرة على الحفاظ على تسجيل الدخول واستعادة التشغيل خلال ثوانٍ دون الحاجة إلى تدخل يدوي.

# نظام حماية الجلسات
أعد تصميم نظام الجلسات ليشمل:

- حفظ تلقائي وآمن للجلسات
- التحقق من سلامة البيانات قبل الحفظ
- منع الكتابة الجزئية أو التالفة
- Atomic Writes
- Session Locking
- Integrity Validation
- Corruption Detection
- Automatic Backup Rotation
- Recovery Snapshots

يجب منع أي عملية Restart من إتلاف الجلسة.

# نظام Restart آمن
قم بإعادة بناء آلية إعادة التشغيل بحيث:

- يتم حفظ الجلسة قبل إعادة التشغيل
- يتم التأكد من اكتمال الكتابة إلى القرص
- يتم إغلاق جميع العمليات بشكل صحيح
- يتم تنظيف الموارد المؤقتة
- يتم استعادة الجلسة بعد الإقلاع مباشرة
- يتم التحقق من صحة الجلسة قبل استخدامها

وامنع:
- تلف ملفات appstate
- الكتابة المتزامنة الخطيرة
- فقدان الكوكيز أثناء Restart
- تسجيل الخروج غير المقصود

# استعادة تلقائية
إذا اكتشف النظام:

- جلسة تالفة
- appstate غير صالح
- ملف مفقود
- بيانات ناقصة

فيجب أن يقوم تلقائيًا بـ:
- استعادة آخر نسخة سليمة
- الرجوع إلى Snapshot سابقة
- إصلاح الملف إن أمكن
- إنشاء تقرير تشخيص

# حماية ملفات الجلسة
قم بتأمين:

- cookies
- appstate.json
- session files
- auth tokens

عبر:
- منع الكتابة غير الآمنة
- منع الحذف العرضي
- التحقق من الصلاحيات
- النسخ الاحتياطي الدوري
- التحقق من سلامة الملفات

# تشخيص مشاكل تسجيل الخروج
قم بتحليل جميع الأسباب المحتملة لـ:

- Invalid AppState
- Expired Sessions
- Cookie Corruption
- Login State Loss
- Concurrent Session Writes
- Messenger Authentication Failures

واكتشف السبب الجذري وأصلحه.

# تحسين Runtime
اجعل إعادة التشغيل:

- أسرع
- أكثر استقرارًا
- غير مؤثرة على الجلسات
- متوافقة مع التشغيل الطويل

# نظام Logs وتشخيص
أضف:

- Session Save Logs
- Session Recovery Logs
- Restart Diagnostics
- Corruption Detection Logs
- Authentication Validation Logs
- Recovery Reports

ويجب أن توضح السجلات سبب أي فقدان للجلسة.

# الاختبار والتحقق
قم بإجراء اختبارات شاملة تشمل:

- إعادة تشغيل متكررة
- إعادة تشغيل أثناء النشاط
- إعادة تشغيل أثناء تنفيذ الأوامر
- إعادة تشغيل أثناء إرسال الوسائط
- إعادة تشغيل بعد ساعات طويلة من التشغيل
- التحقق من استعادة الجلسة بعد كل Restart

ولا تنهِ المهمة حتى يتم التأكد من أن:

- إعادة التشغيل لا تسبب تسجيل خروج
- الكوكيز تبقى سليمة
- appstate يبقى صالحًا
- الجلسات تستعاد تلقائيًا
- لا يحدث أي فقدان للبيانات

# النتائج النهائية المطلوبة

قم بإنشاء:

- نظام Restart احترافي وآمن
- طبقة Session Persistence محسنة
- نظام Recovery تلقائي
- حماية متقدمة للجلسات
- تقرير هندسي كامل يشرح:
  - سبب فقدان الجلسات سابقًا
  - التحسينات المنفذة
  - آلية الاستعادة
  - آلية النسخ الاحتياطي
  - إجراءات الحماية

وقم بإنشاء ملف Markdown عربي يشرح:
- كيفية عمل نظام إعادة التشغيل الجديد
- كيفية استعادة الجلسات
- كيفية إصلاح ملفات appstate
- خطوات الصيانة
- خطوات استكشاف الأخطاء

يجب أن يبدو النظام النهائي وكأنه منصة Messenger احترافية قادرة على إعادة التشغيل آلاف المرات دون فقدان الكوكيز أو الجلسات أو الحاجة إلى إعادة تسجيل الدخول.
