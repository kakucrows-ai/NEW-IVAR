## 🔄 دورة حياة إعادة التشغيل

### قبل إعادة التشغيل (PRE-RESTART)

```
[1] تجميد الكتابات        ← يمنع أي حفظ جديد أثناء الإغلاق
[2] إخطار المجموعة        ← رسالة "جارٍ إعادة التشغيل"
[3] التقاط الحالة         ← api.getAppState() من الاتصال الحي
[4] الحفظ الذري           ← temp → verify → rename
[5] إنشاء لقطة استرداد   ← sessions/appstate_YYYYMMDD_HHMMSS_pre-restart.json
[6] فترة انتظار 1.5 ثانية ← لإتمام مزامنة القرص
[7] process.exit(0)       ← مدير العمليات يُعيد التشغيل تلقائيًا
```

### بعد إعادة التشغيل (POST-RESTART)

```
[1] فحص WAL Journal       ← هل الكتابة الأخيرة اكتملت؟
[2] مسح المصادر           ← primary → backup #1 → ... → backup #5
[3] التحقق من Checksum    ← SHA-256 لكل ملف
[4] التحقق العميق         ← وجود cookies: c_user + xs
[5] استعادة Primary       ← إذا تم التحميل من نسخة احتياطية
[6] تسجيل الدخول          ← nkxfca login() بالحالة المستعادة
```

## 🛡️ الكتابة الذرية (Atomic Write)

### المشكلة القديمة
```js
// ❌ كتابة مباشرة — تُتلف الملف إذا انقطع التيار
fs.writeFileSync("appstate.json", JSON.stringify(data));
```

### الحل الجديد
```
[1] كتابة WAL: { status: "pending" }
[2] كتابة البيانات إلى appstate.json.tmp
[3] قراءة .tmp والتحقق (read-back verification)
[4] كتابة ملف SHA-256
[5] fs.renameSync(.tmp → appstate.json)  ← عملية ذرية
[6] تحديث WAL: { status: "complete" }
```

`renameSync` هي **عملية ذرية** على نفس نظام الملفات — إما يكتمل النقل بالكامل أو لا يحدث شيء.

## 📔 Write-Ahead Log (WAL)

ملف `appstate.json.wal` يُسجّل حالة كل عملية كتابة:

```json
// قبل الكتابة:
{ "v": 2, "status": "pending", "started": 1749349072000, "count": 42 }

// بعد الكتابة الناجحة:
{ "v": 2, "status": "complete", "completed": 1749349072100,
  "count": 42, "checksum": "a1b2c3d4e5f6g7h8" }
```

**عند الإقلاع التالي:**
- إذا وجد `"status": "pending"` → الكتابة الأخيرة انقطعت → تخطّي Primary وتحميل Backup #1
- إذا وجد `"status": "complete"` → كل شيء سليم → تحميل Primary

## 🚨 استكشاف الأخطاء

### الخطأ: `Invalid AppState`

**الأسباب المحتملة:**
1. انتهت صلاحية الكوكيز (expired session)
2. تسجيل الخروج من متصفح آخر
3. قيام Facebook بإلغاء الجلسة أمنيًا
4. فشل استيراد appstate.json

**الحل:**
```
1. افتح لوحة التحكم → 🍪 الجلسة
2. ارفع appstate.json جديد (مستخرج من المتصفح)
3. انتظر إعادة اتصال البوت تلقائيًا
```

---

### الخطأ: `WAL: interrupted write detected`

**المعنى:** انقطع التيار أو أُغلق البوت بالقوة أثناء الحفظ.

**ما يحدث تلقائيًا:**
```
primary مشكوك فيه → تحميل backup #1 → استعادة primary
```
لا يلزم تدخل يدوي.

---

### الخطأ: `ALL SESSION SOURCES FAILED`

**المعنى:** جميع المصادر الستة (primary + 5 backups) تالفة.

**الحل:**
```
1. ارفع appstate.json جديد عبر لوحة التحكم
2. أو استرجع من sessions/ يدويًا:
   cp sessions/appstate_LATEST_*.json appstate.json
3. أعد تشغيل البوت
```

---

### الخطأ: `checksum mismatch`

**المعنى:** محتوى الملف تغيّر بعد حفظه (تلف القرص أو تعديل يدوي).

**ما يحدث تلقائيًا:**
```
تجاهل الملف → الانتقال للنسخة الاحتياطية التالية
```

## 🛠️ الصيانة الدورية

### يوميًا (تلقائي)
- حفظ تلقائي كل 4 دقائق عبر cookieRefresher
- لقطة عند كل إعادة تشغيل

### أسبوعيًا (يدوي)
- تحقق من وجود لقطات في `sessions/`
- تأكد أن البوت متصل ويحفظ الكوكيز

```bash
# فحص حالة الجلسة
ls -la artifacts/messenger-bot/appstate*.json

cat artifacts/messenger-bot/appstate.json.wal
```
